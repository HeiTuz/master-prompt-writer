# 프롬프트 그래프 — 조립 전에 그리는 의존·권한 구조

프롬프트는 문장의 나열이 아니라 **노드와 엣지로 된 구조**다. 좋은 프롬프트가 흔들리는 이유는 대부분 문장이 나빠서가 아니라 **의존 순서가 뒤집혔거나, 같은 축에 권한자가 둘이거나, 답할 수 없는 슬롯이 남아서**다. 평문 템플릿은 이 세 결함을 숨긴다. 그래프는 드러낸다.

MPW에 흩어진 권한 문장들(§4)이 실은 이 그래프다 — 이 파일은 그것을 명시적 구조와 조립 절차로 만든다.

## 1–2. 어휘 — 노드 6종, 엣지 5종

**노드.** `GOAL` 끝난 상태, 루트, 정확히 1개 · `SURFACE` 실행 표면 S1 기계/S2 플랫폼/S3 붙여넣기, 예산·파라미터를 정하는 최상류([image/surfaces.md](image/surfaces.md)) · `SLOT` 채워야 할 값(피사체·팔레트·카피·모션) · `CONSTRAINT` 하드라인·게이트·실패 조건 · `EVIDENCE` 근거(레퍼런스, 스키마 enum, 실측 로그) · `ARTIFACT` 산출 블록, 리프 1개 = 붙여넣기 블록 1개.

`SLOT`↔`CONSTRAINT` 경계: **값을 묻거나 유도할 수 있으면 SLOT(기본값이 있다), 위반 여부만 판정할 수 있으면 CONSTRAINT(기본값이 없다).** "피사체는 여성 1인"은 SLOT, "얼굴을 재드로잉하지 말 것"은 CONSTRAINT.

**엣지.** 판정 기준을 통과해야 엣지를 그린다. 못 하면 그냥 두 노드다. 위반 증상은 §7.

| 엣지 | 이 엣지인지 판정하는 법 |
|---|---|
| `requires` | A가 미정인 채로 B를 **쓸 수 없으면** requires. 일단 쓰고 나중에 고치면 되는 정도면 아니다 |
| `authority` | 둘이 **같은 축에** 값을 주면 authority이고, 이기는 쪽이 A다. 축이 다르면 별개 슬롯 둘이다 |
| `derives` | A만 보고 B를 **적어낼 수 있으면** derives(A가 B의 기본값을 준다). 물어봐야만 정해지면 아니다 |
| `excludes` | 둘 다 지킨 산출물을 **하나도 상상할 수 없으면** excludes. 어색할 뿐이면 아니다 |
| `feeds` | A의 산출물이 **실행 단위 경계**(다른 호출·다른 레인)를 건너 B의 입력이 되면 feeds |

`requires`↔`feeds` 경계: **경계를 건너 넘겨주는 것이 있으면 `feeds`, 한 블록 안에서 쓰는 순서만 정하면 `requires`.** `feeds`에만 인터페이스(§5)를 붙인다.

`authority`는 **방향이 있어야 하고 사이클이 없어야 한다.** 사이클은 프롬프트 문제가 아니라 요구사항 문제다 — 쓰지 말고 질문을 낸다.

## 3. 조립 절차 — 그래프에서 프롬프트로

1. **노드 수집.** 요청·증거·표면에서 GOAL·SURFACE·SLOT·CONSTRAINT·EVIDENCE를 뽑는다.
2. **SURFACE를 먼저 고정한다.** 다른 모든 노드의 상류다.
3. **`requires`로 채우는 순서를 정한다.** "A 없이는 B를 못 쓴다"를 전부 만족하는 순서를 하나 만든다. 만들 수 없으면(A가 B를, B가 A를 요구) 요구사항 충돌이다 — 쓰지 말고 질문을 낸다.
4. **`authority` 사이클 검사.** 축마다 권한자가 정확히 하나인지 본다(§4).
5. **`derives`로 빈 슬롯을 먼저 채운다.** 유도 가능한 값은 묻지 않는다.
6. **남은 미결 슬롯 중 결과·하드라인을 바꾸는 것만 질문한다**(≤3). 나머지는 저위험 기본값 + `가정:` 한 줄.
7. **`excludes` 위반 검사.** 걸리면 우선순위 한 줄로 해소하거나 컷을 분리한다.
8. **직렬화.** 3에서 정한 순서대로 쓰되, 표면이 파라미터를 갖는 축은 산문에서 뺀다([image/surfaces.md](image/surfaces.md) §4).
9. **리프 = 블록.** 리프가 둘이면 블록도 둘이다.

## 4. 축별 단일 권한 — 한 축에 권한자는 하나

권한이 둘이면 모델이 고른다. 그게 곧 재현 불가다.

| 축 | 권한 노드 | 지는 것 |
|---|---|---|
| 표면·예산·비율·해상도 | `SURFACE` | 문서의 관례값, 과거 프롬프트의 값 |
| 구조·레이아웃 | 패턴 노드 1개(P 하나 / TP 하나 / 포맷 A·B 중 하나) | 다른 패턴의 구조 어휘 |
| 팔레트 | 팔레트 권한자 1개(P 또는 L 또는 TP, 파라미터가 있으면 파라미터) | 본문에 흩어진 색 언급 |
| 정체성 | 레퍼런스/`soul_id` | 텍스트 얼굴 묘사 |
| 의복 구조 | 관측 증거 | 미화 형용사 |
| 카메라·렌즈 | 바디 1 · 주 렌즈 1 | 추가 렌즈·장비 스펙 |
| 조명 | 광원 방향·질 1세트 | 장비명, 중복 조명 서술 |
| 텍스트 카피 | 롤 라벨 블록 1곳 | 본문에 떠도는 같은 카피 |
| 안전 | 명시 선언된 티어 | 휴리스틱 승격 |
| 완료 판정 | `prime` / DoD | worker의 완료 주장 |

세로로 읽으면 "무엇이 이기는가", 가로로 읽으면 "무엇을 지워야 하는가"다.

## 5. 체인 — `feeds` 엣지의 계약

A의 산출이 B의 입력이면 엣지에 **인터페이스**를 붙인다. 붙이지 않은 `feeds`는 반드시 중간에서 깨진다.

각 `feeds` 엣지가 답해야 할 것: ① 넘어가는 것의 형식 ② 넘어가지 않는 것(초안·중간 사고·원본 경로) ③ B가 A의 산출을 받았는지 판정하는 방법.

예: 레퍼런스 분석 → 프롬프트 컴파일 체인에서는 관측 사실만 넘기고 원본 파일 경로는 넘기지 않는다. 컴파일 → 실행 체인에서는 완성 프롬프트 1개만 넘기고 거친 원문은 넘기지 않는다([image/compiler.md](image/compiler.md) §1-5).

## 6. TEAM·GOAL 모드 — 같은 그래프, 노드가 작업일 뿐

멀티에이전트에서는 SLOT 대신 **작업**이 노드이고 엣지 규칙은 그대로다. 작업 노드에서는 그 규칙이 join gate·validation-coupled decomposition으로 나타난다 — 정본은 [model-playbooks.md](model-playbooks.md)이며 여기서 다시 정의하지 않는다.

## 7. 실패 증상 → 그래프 결함 역추적

| 증상 | 그래프 결함 | 고치는 법 |
|---|---|---|
| 같은 프롬프트인데 결과가 매번 다름 | 한 축에 `authority` 둘 | §4로 권한자 1개만 남기고 나머지 삭제 |
| 뒤 문장이 앞 지시를 무효화 | `requires` 순서 뒤집힘 | §3-3에서 정한 채움 순서로 재배치 |
| 질문이 너무 많음 | `derives` 미사용 | 유도 가능한 값을 기본값으로 채우고 `가정:`으로 고지 |
| 모델이 지시를 임의 해석 | `excludes` 미해소 | 우선순위 한 줄 추가 또는 컷 분리 |
| 넣은 레퍼런스가 결과에 안 보임 | 고아 `EVIDENCE`(엣지 없음) | 어느 SLOT을 잠그는지 연결하거나 빼라 |
| 체인 중간에서 형식이 깨짐 | `feeds` 인터페이스 없음 | §5의 3가지를 엣지에 붙임 |
| 블록이 길이를 넘김 | 리프 하나에 ARTIFACT 둘 | 리프를 쪼개 블록 분리 |
| "다 했다"는데 판정 불가 | `authority` 루트 부재 | DoD를 prime 노드에 붙임 |

## 8. 언제 쓰지 않는가

한 문장이면 끝나는 요청에 그래프를 그리지 않는다. 노드가 3개 이하이고 충돌 축이 없으면 그냥 쓴다. **그래프를 산출물에 그려서 보여주지도 않는다** — 그래프는 조립 도구이지 납품물이 아니다. 사용자에게 나가는 것은 완성된 프롬프트 블록이다.

그래프를 실제로 그릴 값어치가 있는 신호: 산출물이 2개 이상, 체인이 있음, 레퍼런스·증거가 있음, 같은 축에 지시가 둘 이상 들어옴, 멀티에이전트, 또는 앞서 만든 프롬프트가 재현되지 않음.
