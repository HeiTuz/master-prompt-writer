# 실행 표면 계약 (S1·S2·S3) — IMAGE·COMPOSITE·영상 공통 선행 판정

**이 파일이 길이·비율·해상도·품질·파라미터 소관의 정본이다.** 다른 파일의 사이즈·길이 문장이 이 파일과 충돌하면 이 파일이 이긴다.

프롬프트를 쓰기 전에 **어느 표면으로 나가는지**부터 정한다. 같은 모델이라도 표면이 다르면 계약이 다르다. 예: `gpt-image-2`는 기계 핸드오프(S1)에서 픽셀 `size` 필드를 갖고, Higgsfield MCP(S2)에서는 픽셀 개념 없이 `aspect_ratio` + `resolution` 티어를 갖는다. 표면을 정하지 않고 쓴 규칙은 둘 다 틀린다.

## 0. 표면 판정 — 한 줄 질문

| 질문 | 답 | 표면 |
|---|---|---|
| 산출물이 JSON 계약으로 하류 실행기에 넘어가나? | 예 | **S1 기계 핸드오프** |
| 실행기가 모델 id와 파라미터를 받는 플랫폼 API/MCP인가? | 예 | **S2 플랫폼 파라미터** |
| 사람이 웹/앱 입력창에 직접 붙여넣나? | 예 | **S3 붙여넣기 UI** |

미지정이면 S3로 가정하고 한 줄로 고지한다. S3 가정은 가장 좁은 계약이므로 나중에 S1·S2로 승격해도 깨지지 않는다.

## 1. S1 — 기계 핸드오프 (MPW → ImgGen2 계약 경로)

**정본은 문서가 아니라 스키마다.** `contracts/v1/production-adapter-options.schema.json`, `contracts/v1/imggen2-production-record.schema.json`이 유일한 권위이며, 이 파일은 스키마 값을 복제하지 않는다. 값이 궁금하면 스키마를 읽는다.

- 허용 `ar`·`size`·`quality`·`output_format`은 **스키마 enum이 전부**다. 문서 어딘가에서 본 비율·픽셀값이 스키마에 없으면 그 값은 쓸 수 없다.
- `ar`↔`size` 매핑은 `contracts/validate.py`의 `GEOMETRY` 표가 판정한다. 불일치는 `production_geometry_mismatch`로 거부된다.
- 길이 계약: 스키마 필드 제약이 하드라인이다. `prompt-bundle/v1`의 블록은 `text.maxLength: 2000` + `unicode_char_count ≤ 2000`을 **계약으로** 갖는다(2026-07-25 스키마 직접 확인) — 같은 숫자지만 근거가 붙여넣기 UX가 아니라 스키마라는 점이 다르다. 다른 필드의 상한(경로 500 등)도 스키마가 정한다.
- 컴파일 전 `python3 contracts/validate.py`로 실제 검증한다. 문서 대조로 갈음하지 않는다.

**금지:** 스키마에 없는 비율·픽셀·품질값을 "문서에 있으니 된다"고 산출하는 것. 스키마 enum 밖 값은 컴파일 실패로 처리한다.

## 2. S2 — 플랫폼 파라미터 (Higgsfield MCP 등)

**모델 id를 먼저 고르고, 그 모델의 `parameters` / `aspect_ratios` / `medias.roles`가 계약이다.** 모델 후보 선택은 [model-routing.md](model-routing.md)를 따르고, 실제 파라미터는 런타임에서 확인한다.

- **런타임 확인이 문서보다 우선한다.** `models_explore(action:"get", model_id:...)`가 반환한 파라미터가 정본이다. [model-routing.md](model-routing.md)의 스냅샷은 후보를 좁히는 용도이지 파라미터 근거가 아니다.
- **픽셀 size를 만들어내지 않는다.** **이미지 생성 모델의 해상도 축은 티어다** — 모델별 `resolution`(`1k`/`2k`/`4k` 등) 또는 `quality`(`low`/`medium`/`high`, `basic`/`high`, `1.5k`/`2k`). 이 모델들에 `1024x1536` 같은 값을 프롬프트나 파라미터로 지어내지 않는다.
- **다만 "S2에는 픽셀 개념이 없다"는 단정은 틀리다.** 일부 영상·후처리·스프라이트 모델은 명시적 픽셀 파라미터를 갖는다: `marketing_studio_video`의 `width`/`height`, `clipify`의 `max_height`(144~2160), `autosprite`의 `frame_size`(32~512)(2026-07-25 런타임 확인). **그 모델의 런타임 정의를 따른다** — 픽셀 파라미터가 정의돼 있으면 티어로 우회하지 말고 그 파라미터를 쓰고, 없으면 지어내지 않는다. 판정 기준은 표면이 아니라 그 모델의 `parameters`다.
- **비율은 모델별로 다르다.** 같은 `4:5`도 어떤 모델엔 있고 어떤 모델엔 없다. 모델의 `aspect_ratios` 배열 밖 값을 쓰지 않으며, `auto`가 배열에 있으면 유효한 선택지다(전역 `auto` 금지 규칙은 S1 한정이다).
- **`aspect_ratios`가 빈 배열이면 그 모델은 비율을 파라미터로 받지 않는다 — 비율을 넘기지 않고 입력 이미지·모델 기본값을 따른다.** "배열 밖 값 금지"를 "아무 값도 못 쓴다"로 읽어 모델을 교체하는 것도, 없는 파라미터를 억지로 만들어 넣는 것도 오답이다. 실제 빈 배열 사례: `minimax_hailuo`, `autosprite`, `clipify`, `tripo_3d`(2026-07-25 런타임 확인). 비율이 레인의 하드 요구인데 후보가 빈 배열이면 그때만 모델을 바꾸거나 후처리 `reframe`으로 처리한다.
- **레퍼런스는 `medias.roles`로 전달한다.** `image`·`image_references`·`start_image`·`end_image`·`video_references`·`audio_references`는 역할이 다르다. 슬롯이 전달하는 것을 프롬프트에서 다시 묘사하지 않는다.
- 길이 상한은 계약으로 존재하지 않는다. 대신 §4 신호 밀도 규칙을 적용한다.

## 3. S3 — 붙여넣기 UI (사람이 직접 입력창에 붙여넣음)

파라미터가 없거나 사용자가 UI에서 따로 고른다. 프롬프트가 모든 것을 흡수해야 한다.

- **블록당 공백 포함 2000자 하드라인. 이 표면에서만 하드라인이다.** 내보내기 전 `wc -m`으로 실측하고 `길이:` 노트에 실측값만 쓴다.
- 경로 참조 금지, 자기완결. 초과하면 장식 → 중복 → 방법 설명 순으로 줄이고, 그래도 넘으면 컷을 분리한다.
- UI에서 고르는 값(비율·프리셋·품질·길이)은 프롬프트 본문이 아니라 **별도 라벨 한 줄**로 전달한다.

## 4. 파라미터 우선 규칙 (S1·S2 공통)

**파라미터로 표현되는 축은 산문에 중복 기술하지 않는다.** 중복은 토큰만 쓰고, 파라미터와 산문이 어긋나면 결과가 흔들린다. 이는 [../model-playbooks.md](../model-playbooks.md) §공통 배치 규칙의 "런타임 레버가 있으면 프롬프트로 우회하지 않는다"를 이미지·영상 레인에 적용한 것이다.

| 축 | 파라미터가 있으면 | 산문에는 |
|---|---|---|
| 비율 | `aspect_ratio` / `ar` | 쓰지 않는다 (S3만 끝에 `AR x:y`) |
| 해상도·품질 | `resolution` / `quality` / `mode` | 쓰지 않는다 |
| 팔레트 | `colors[]` / `background_color` (예: `recraft_v4_1`) | 파라미터가 없을 때만 HEX를 본문에 |
| 영상 길이 | `duration` (모델별 **열거값 또는 범위**) | 쓰지 않는다. 임의 초 지정 금지 |
| 오디오 | `generate_audio` / `sound` | 대사·SFX **내용**만. on/off는 파라미터 |
| 장르 톤 | `genre` — **enum이 모델마다 다르다.** 값을 쓰기 전에 그 모델의 옵션을 런타임 확인한다 | 장르 라벨 대신 화면 결과만 |
| 속도 효과 | `speedramp` / `slow_motion` | 쓰지 않는다 |
| 멀티샷 | `multi_shots` + `multi_prompt` | 샷 경계를 산문으로 흉내내지 않는다 |
| 프롬프트 준수 | `cfg_scale` | "지시를 정확히 따르라" 같은 메타 지시 금지 |
| 시작·끝 프레임 | `start_image` / `end_image` 롤 | 입력 이미지를 재묘사하지 않는다 |
| 프리셋 | `preset_id` (**image-to-video 전용**) | 프리셋 이름을 본문에 쓰지 않는다 |
| 스타일·브랜드킷 | `style_id` / `brand_kit_id` / `product_ids` | 로고·색 규격을 본문에 복제하지 않는다 |

**산문이 소유하는 것:** 장면·피사체·행동·공간, 조명의 방향과 화면 결과, 카메라 시점·거리, 질감·소재, 렌더될 정확 카피, 실패 조건.

**반례 처리:** 파라미터를 넣었는데 결과가 무시하면, 산문을 덧붙이기 전에 ① 파라미터가 그 모델에 실제로 존재하는지 ② 값이 허용 enum 안인지를 먼저 확인한다. 둘 다 통과했는데도 무시되면 그때만 산문 1회 보강을 허용하고, 그 사실을 dated note로 남긴다.

## 5. 후처리는 별도 단계다 — 생성 프롬프트의 실패로 취급하지 않는다

배경 제거·아웃페인트·업스케일·리프레임·립싱크는 전용 도구가 있다. 생성 프롬프트로 흉내내거나 "이 레인의 비목표"로 잘라내지 말고 **후속 단계로 명시**한다.

| 필요 | 처리 |
|---|---|
| 투명 배경 / 컷아웃 | 생성 → `remove_background`(`image_background_remover`). 생성 단계는 단색 배경으로 설계 |
| 캔버스 확장 | `outpaint` |
| 해상도 상향 | `topaz_image` / `bytedance_image_upscale` (영상은 `topaz_video` / `bytedance_video_upscale`) |
| 영상 비율 변경 | `reframe` |
| 벡터·SVG 계열 자산 | 후처리가 아니라 **모델 선택** — `recraft_v4_1` `model_type: vector` |

생성 후 코드로 글자를 얹는 합성은 여전히 금지다([compiler.md](compiler.md) 철칙 9). 위 항목은 글자 합성이 아니라 이미지 연산이므로 별개다.
