# from-image 레인 — 이미지가 입력으로 들어올 때

이미지를 관측해서 프롬프트를 만드는 두 진입점(**취향 레퍼런스** / **생성물 재컴파일**)을 한 레인으로 다룬다. 둘은 **같은 관찰 레코드를 쓰고, 권한 배선과 변주 다이얼만 다르다.** 그래서 파일을 나누지 않는다.

**이 파일이 정본인 것:** 관찰 레코드의 축 정의(§3), 생성물 재컴파일 함정(§4), 감도 축 ↔ 정체 축 분리와 변주 다이얼·복제 판정(§5), 관찰 → 타깃별 컴파일의 `feeds` 계약(§6).

**이 파일이 정본이 아닌 것 — 참조만 한다:**

| 소관 | 정본 |
|---|---|
| 길이·비율·해상도·품질·파라미터 | [surfaces.md](surfaces.md) |
| 축별 단일 권한·엣지 판정·체인 | [../prompt-graph.md](../prompt-graph.md) |
| 목적축 → 모델 후보 | [model-routing.md](model-routing.md) |
| 네거티브 티어·철칙 10 | [compiler.md](compiler.md) §2 |
| 레인 게이트 카드·이미지 슬롯 기본값 | [lanes.md](lanes.md) |
| 의상 관찰 스키마·파생물 | `~/src/higgsfield-prompt-bridge/references/fashion-vision-analysis-contract.md` + `scripts/fashion_analysis.py` |
| 비전 반환형 타입 안전 | `~/src/higgsfield-prompt-bridge/references/vision-result-type-safety.md` |

§5의 임계값(축 개수, 다이얼 3단 명칭)은 **운영 설계값이고 법적 기준이 아니다.** 법원은 축을 세지 않는다. 상업 캠페인·분쟁 리스크가 있는 건은 변호사 검토가 별도로 필요하다.

## 1. 진입점 2종 — 무엇이 갈리는가

| | **취향 레퍼런스** | **생성물 재컴파일** |
|---|---|---|
| 입력 | 무드보드·화보·스크랩 | 내가 만든 이미지 |
| 목적 | 같은 감도로 **다른** 결과 | 재현·반복·다른 플랫폼으로 이식 |
| 레퍼런스가 갖는 권한 | **감도 축(S1~S4)만.** 정체 축은 권한 없음 | **전 축.** 단 `artifact` 태그 값은 제외 |
| 변주 다이얼 기본값 | **D2 자매 컷** (§5.4) | **D1 인접 변주** (§5.4) |
| 원래 프롬프트 존재 여부 | 묻지 않는다 | **먼저 묻는다** (§4.4) |

**공유하는 것:** §2 비전 경계, §3 관찰 레코드, §6 컴파일 타깃, §7 게이트.

### 1.1 진입점 판정 — 한 줄 질문

> **이 이미지를 만든 것이 사용자 자신인가?**
> 예 → 생성물 재컴파일. 아니오/불명 → 취향 레퍼런스.

불명이면 취향 레퍼런스로 간다. 좁은 쪽 계약이므로 나중에 승격해도 깨지지 않는다.

### 1.2 혼합 입력 — 레퍼런스가 여럿일 때

레퍼런스가 여럿이면 **레퍼런스마다 authority 축을 따로 배정한다.** 한 축에 레퍼런스가 둘이면 그 축은 재현되지 않는다([prompt-graph.md](../prompt-graph.md) §7 "결과가 매번 다름").

- 배선 예: `ref-01`(내 생성 인물컷) = I9 얼굴 기하 + I6 의상 / `ref-02`(외부 무드보드) = S1~S4 감도.
- 충돌하면 산출 첫 문장에 **우선순위 1줄**을 박는다. 관련 함정 정본: [image-reference-editing-prompt-pitfalls.md](../image-reference-editing-prompt-pitfalls.md), [image-reference-wardrobe-preservation-pitfall.md](../image-reference-wardrobe-preservation-pitfall.md).
- 어떤 축의 권한자도 아닌 레퍼런스는 **빼거나 축을 배정한다.** 넣어놓고 안 쓰는 것은 고아 EVIDENCE다.

## 2. 비전 런타임 중립 — 반환형이 달라도 레코드는 하나

런타임마다 반환형이 다르다. **두 경로를 모두 지원하고, 어느 쪽이든 도착점은 §3 관찰 레코드다.**

| 경로 | 상황 | 처리 |
|---|---|---|
| **A. 도구 반환** | 비전 도구가 텍스트·객체·리스트를 반환 | 원시 payload를 문자열 처리기에 넘기지 않는다. 값이 텍스트인지 확인한 뒤에만 정규식·직렬화를 건다 |
| **B. 네이티브 비전** | 이미지가 컨텍스트에 붙고 에이전트가 직접 읽음. 도구는 확인 응답만 반환하거나 아예 없음 | 에이전트가 픽셀을 읽어 레코드를 **직접 작성한다.** 도구 응답 부재를 실패로 처리하지 않는다 |

세부 계약은 재정의하지 않는다 — `vision-result-type-safety.md`가 정본이다. 이 레인이 그 위에 얹는 것은 아래 넷뿐이다.

1. **라벨은 `ref-01`, `ref-02` …** 첨부 순서를 보존한다. 산출 블록에는 이 라벨도 원본 파일명도 넣지 않는다(§6.4).
2. **레코드가 없으면 컴파일하지 않는다.** "이미지를 봤다"는 진술은 레코드가 아니다. 축마다 값 또는 상태가 있어야 레코드다.
3. **단일 이미지는 리스트를 만들지 않는다.** 1항목 리스트를 만들었다가 문자열로 강제 변환하는 경로가 실패 원인이다.
4. **타입 실패 후 재시도 금지.** 같은 호출을 반복해 같은 원시값을 같은 포매터에 다시 넣지 않는다. 이미 로드된 이미지를 그대로 두고 수동으로 레코드를 써서 같은 턴에 진행한다. 예외 메시지를 산출 프롬프트 자리에 대신 넣지 않는다.

## 3. 관찰 레코드 — 플랫폼 중립

### 3.1 한 줄의 형식

```text
<ref 라벨> | <축> | <상태> <confidence> | <값 — 결과 어휘 + 수치> | 근거: <픽셀 단서>
```

예:

```text
ref-01 | S1 조명 | observed 0.8 | 좌 45° 창광, 소프트, 키:그림자 약 2:1, 그림자 가장자리 부드러움 | 근거: 좌뺨 하이라이트 + 우측 코그림자 경계 폭
ref-01 | I6 의상 아이템 | observed 0.6 | 오버사이즈 더블 브레스트, 노치트 라펠, 무릎 아래 길이 | 근거: 라펠 절개선·단추 2열
ref-01 | I8 렌더 텍스트 | unreadable 0.9 | 좌상단에 4~6글자 상당 문자열, 글리프 불성립 | 근거: 자획이 닫히지 않음
```

### 3.2 상태 5종 — 승격 금지

| 상태 | 뜻 | 컴파일에 실리는가 |
|---|---|---|
| `observed` | 픽셀에서 직접 읽음 | 실린다 |
| `inferred` | 다른 관찰에서 추론 | 실린다. 단 confidence를 함께 남긴다 |
| `unreadable` | 있는 건 보이나 판독 불가 | 실리지 않는다. 값을 만들지 않는다 |
| `not_visible` | 프레임·가림 때문에 안 보임 | 실리지 않는다 |
| `not_applicable` | 그 축이 이 이미지에 없음 | 실리지 않는다 |

**보이지 않는 것을 발명하지 않는다.** 뒤판·안감·잠금장치·브랜드·소재 혼용률·정확 치수는 안 보이면 안 보이는 것이다. `unreadable`/`not_visible`을 `observed`로 올리는 순간 그 값은 근거 없는 지시가 되고, 모델은 그것을 사실로 렌더한다.

### 3.3 축 목록 — 게이트와 같은 축을 쓴다

레코드의 축은 §5 게이트의 축과 **동일하다.** 그래야 결과 판정이 기계적으로 돌아간다.

**감도 축 — 가져와도 "같은 감도"**

| 축 | 기록할 성분 |
|---|---|
| **S1 조명** | 광원 방향(각도), soft/hard, 키:그림자 비율, 그림자 가장자리 성질, rim 유무, 켈빈 |
| **S2 카메라** | 카메라-피사체 거리, 얼굴·평면 압축 정도, 심도 깊이, 배경 분리 정도, 샷 사이즈 대역 |
| **S3 그레이딩** | 채도 대역, 콘트라스트 커브, 색온도 편향, shadow/highlight 롤오프, 그레인 양, 노출 편향 |
| **S4 프레임 정책** | 네거티브 스페이스 비율(%), 정면성 vs 스냅, 대칭 선호, 헤드룸, 피사체 크기 비중 |

**정체 축 — 가져오면 "복제"**

| 축 | 기록할 성분 |
|---|---|
| **I1 포즈** | 팔·다리 관절 각도 조합, 머리 기울기, 무게중심 |
| **I2 시선·표정** | 시선 방향 + 표정의 특정 조합 |
| **I3 손** | 손 위치·제스처·소지물 |
| **I4 소품** | 소품 종류 + 프레임 내 배치 |
| **I5 공간 정체** | 그 벽·그 건축 요소·랜드마크·실재 매장 |
| **I6 의상 아이템 정체** | 그 실루엣의 그 아이템 |
| **I7 크롭·배치** | 피사체 프레임 내 좌표, 눈이 오는 분할선 |
| **I8 렌더 텍스트** | 카피 문자열, 타이포 배치 (기본 상태는 §4.3) |
| **I9 얼굴 기하** | 페이스 랜드마크 |

### 3.4 결과 어휘 + 수치 — 형용사는 레코드가 아니다

`감성적인`, `고급스러운`, `세련된`은 화면 결과를 아무것도 정하지 못하므로 모델이 평균값을 낸다([compiler.md](compiler.md) 철칙 3). 감도 축은 **방향·비율·거리·켈빈·채도 대역·여백 %** 로 적는다. 장비명은 결과로 환원한다(철칙 4). 필름·룩 이름은 화면 결과와 병기한다(철칙 10).

### 3.5 의상이 주제면 위임한다 — 여기서 스키마를 다시 쓰지 않는다

| 상황 | 처리 |
|---|---|
| 의상이 결과의 하드락(그 아이템을 정확히 재현해야 함) | `fashion-vision-analysis-contract.md` + `scripts/fashion_analysis.py`로 **위임**. 반환된 `reference_authority_map`·`wardrobe_hard_locks`·`unknown_warnings`를 그대로 쓴다 |
| 의상이 감도 성분만(실루엣 계열·소재 질감 정책) | I6 한 줄로 충분. 위임하지 않는다 |

위임 경로에서 온 값을 이 파일의 축 표로 옮겨 적지 않는다. 두 곳에 같은 값이 있으면 드리프트가 시작된다.

## 4. 생성물 재컴파일 — 이 진입점에만 있는 함정

취향 레퍼런스에는 없고 **내가 만든 이미지를 다시 프롬프트로 되돌릴 때만** 생기는 결함 네 종이다.

### 4.1 값마다 3태그 — `intent` / `signature` / `artifact`

생성물 재컴파일에서는 §3.1 레코드 한 줄에 태그를 하나 더 붙인다.

| 태그 | 판정 질문 | 컴파일 |
|---|---|---|
| `intent` | 내가 프롬프트로 지시했던 것인가 | 그대로 실린다 |
| `signature` | 그 모델이 지시 없이도 내는 성질인가 | §4.2 분기 |
| `artifact` | 물리적으로 성립하지 않는가 | **싣지 않는다** |

출처가 생성물인지 불명이어도 **artifact 스캔은 그대로 돌린다** — 아티팩트 판정은 출처 지식 없이 픽셀만으로 가능하다. `signature` 판정은 모델을 알아야 가능하므로 모르면 `inferred` + 낮은 confidence로 둔다.

### 4.2 AI 아티팩트 — 충실한 관찰이 다음 생성에서 재현된다

관찰이 정직할수록 결함이 정확히 프롬프트에 실리고, 다음 생성이 그걸 그대로 재현한다. **이게 재컴파일 최대 실패 모드다.**

판정 가능한 `artifact` 목록:

- 손가락 수·관절 방향이 성립하지 않음
- 글리프가 닫히지 않는 문자, 유령 글자, 중복 문자열
- 이어지지 않는 봉제선·스트랩·지퍼, 사라지는 소매
- 비대칭 액세서리(귀걸이 한쪽, 안경 다리 한쪽)
- 반복 패턴의 어긋난 이음매, 배경 격자·타일의 끊김
- 접지 그림자 없는 발, 그림자 방향이 광원과 어긋남
- 얼굴·신체 일부만 다른 해상도로 뭉개짐

규칙 두 줄:

1. **레코드에는 남기고 프롬프트에서는 뺀다.** 남겨야 "왜 안 실었는지"가 추적된다. 재현이 목적이어도 예외를 두지 않는다 — 결함 재현은 재현의 목표가 아니다.
2. **아티팩트를 부정문으로 배제하지도 않는다.** `손가락 6개 없음`은 부정문이고, 애초에 손가락 개수는 프롬프트 축이 아니다. 침묵이 정답이다. 손·글자 결함의 일반 예방은 [realism.md](realism.md)와 [compiler.md](compiler.md) §2 소관이다.

### 4.3 모델 서명 — 재서술하면 이중 적용된다

결과물에는 이미 그 모델의 서명이 얹혀 있다. 그걸 관찰해서 프롬프트에 다시 쓰면 **다음 생성에서 두 번 적용된다** — 보케가 더 무너지고, 피부가 더 미끄럽고, 색온도가 더 치우친다.

판정 질문: **"이 성질이 그 모델의 무프롬프트 산출에도 나타나는가?"** 예면 `signature`.

전형적 서명: 과한 보케·배경 붕괴, 특유의 피부 광택·과매끈, 기본 색온도 편향, 중앙 정렬 구도 편향, 과채도, 얼굴 대칭화, 기본 시네마틱 그레이드.

| 다음 생성 타깃 | `signature` 처리 |
|---|---|
| **같은 모델로 다시 돌린다** | 프롬프트에 **쓰지 않는다.** 자동으로 온다. 억제하려면 반대 방향 결과 어휘를 쓴다(예: 배경 분리를 줄이려면 심도 깊이를 결과로 지정) |
| **다른 모델로 이식한다** | 결과 어휘로 **명시적으로 재서술한다.** 이때만 싣는다 |

서명인지 의도인지 확신이 없으면 이식 쪽에만 싣는다(보수적). 같은 모델 재실행에서 빠진 것은 어차피 다시 나오지만, 이식에서 빠진 것은 사라진다.

### 4.4 렌더 텍스트 — 기본 상태는 `unreadable`

생성 이미지의 글자는 대개 깨져 있다. 관찰이 깨진 문자열을 그대로 전사하면 그 문자열이 다음 생성의 카피가 된다.

- **I8의 기본 상태는 `unreadable`이다.** 읽히더라도 의도한 문구는 사용자에게 확인한다.
- 확인 전에는 카피를 싣지 않는다. `문자 없는 클린 마감`으로 두거나 롤 라벨 자리를 비워둔다.
- **카피의 권한자는 사용자이지 관찰이 아니다.** 카피는 롤 라벨 블록 1곳에만 둔다([prompt-graph.md](../prompt-graph.md) §4).
- 타이포 배치(레이아웃 좌표)는 카피와 별개 축이다. 배치는 `observed`일 수 있고 문자열은 `unreadable`일 수 있다.

### 4.5 원래 프롬프트가 있으면 그게 권한자다

재현이 목적이면 **관찰을 시작하기 전에 원래 프롬프트가 있는지 묻는다.**

| 있음 | 없음 |
|---|---|
| 원래 프롬프트·모델 id·파라미터·seed가 **권한자**. 관찰은 델타 확인용으로만 쓴다 | 관찰이 권한자. 관찰로 복원할 수 없는 축(seed, 내부 샘플링)은 **재현 대상에서 뺀다** |

- **완전 동일 재현을 약속하지 않는다.** seed가 없으면 같은 프롬프트도 같은 이미지를 주지 않는다.
- **다른 플랫폼으로 이식할 때는 원래 프롬프트를 그대로 옮기지 않는다.** 모델을 바꾸면 프롬프트도 바꾼다([model-routing.md](model-routing.md) §3 규칙 3). 원래 프롬프트는 의도의 근거로 쓰고, 문법·어휘·구조는 새 타깃 계약으로 다시 쓴다.
- 원래 프롬프트와 관찰이 어긋나면 **관찰이 실제로 나온 화면**이다. 어긋난 지점을 사용자에게 알리고, 어느 쪽을 재현할지 묻는다.

## 5. 복제 회피 계약 — 감도는 남기고 정체는 버린다

> 이 절의 축 개수 임계값과 다이얼 3단 명칭은 **운영 설계값**이다. 실산출물로 캘리브레이션이 필요하고, 법적 판단 기준이 아니다.

### 5.1 재정의 — "얼마나 베끼나"가 아니라 "권한 축이 몇 개인가"

레퍼런스는 프롬프트 그래프의 `EVIDENCE` 노드다. [prompt-graph.md](../prompt-graph.md) §4는 한 축에 권한자 하나를 요구하고 §7은 엣지 없는 EVIDENCE를 결함으로 본다. 둘을 합치면:

- **감도 소실** = 레퍼런스가 아무 축의 권한자도 아님(고아 EVIDENCE).
- **복제** = 레퍼런스가 너무 많은 축의 권한자임.
- **원하는 상태** = 레퍼런스가 정해진 소수 축에서만 권한자.

조작 가능한 형태는 하나다: **레퍼런스에 authority 엣지를 몇 개, 어느 축에 그을 것인가.**

### 5.2 감도 축 vs 정체 축 — 판정 질문 2개

| 질문 | 답 → 판정 |
|---|---|
| **Q-재적용** — 이 축의 값을 다른 피사체·다른 날·다른 세트에 그대로 적용해도 말이 되는가 | 된다 → **감도 축** / 그 컷이 아니면 말이 안 된다 → **정체 축** |
| **Q-지목** — 제3자가 이 축 하나만 보고 원본을 지목할 수 있는가 | 지목 가능 → **정체 축** |

두 질문이 엇갈리면 **정체 쪽으로 센다**(보수적).

축 목록은 §3.3과 같다. 감도 = S1~S4, 정체 = I1~I9.

**팔레트의 특수 처리:** 레퍼런스에서 가져오는 것은 **관계**(채도 대역, 명도 스프레드, 60/30/10 비중, 색상환 간격)이고, **HEX 값 자체는 그 관계를 만족하는 새 값으로 재생성한다.** D1(자기 자산)만 예외다. 반면 조명 수치(키:그림자 2:1, 5600K)는 물리 설정이지 정체가 아니므로 그대로 가져와도 된다. 실행자가 팔레트 파라미터를 가지면 그쪽이 권한자이고 재생성된 HEX는 산문에 중복하지 않는다([surfaces.md](surfaces.md) §4).

### 5.3 사용자 5축(스타일·의상·컬러·포즈·헤어)과의 대응

5축은 감도/정체 경계와 **직교하지 않는다.** 축 이름 하나가 감도 성분과 정체 성분을 같이 문다.

| 사용자 축 | 감도 성분 | 정체 성분 |
|---|---|---|
| 스타일 | 장르 태도·편집 태도, 스타일 라벨의 조명·그레이딩 기본값 | 그 컷의 특정 연출 |
| 의상 | 실루엣 계열, 소재 질감 정책 | 개별 아이템 정체(I6) |
| 컬러 | 채도 대역·콘트라스트·팔레트 관계 | HEX 세트 통짜 |
| 포즈 | 자세 계열, 에너지 | 관절 각도 조합(I1) |
| 헤어 | 길이·질감 대역 | 특정 스타일링 인스턴스 |

**핵심: 5축에는 감도의 주 담지자가 없다.** S1 조명·S2 카메라·S3 그레이딩·S4 프레임 정책 중 어느 것도 5축에 명시적으로 없다. "컬러"가 S3의 일부를 흡수하고 "스타일"이 나머지를 뭉뚱그릴 뿐이다.

**"최소 몇 축을 고정해야 감도가 남는가"는 잘못된 질문이다.** 올바른 형태:

- **감도 앵커 4축(S1~S4)이 고정되어 있는가.** 고정이면 사용자 5축을 **전부** 바꿔도 감도는 남는다.
- S1~S4가 풀리면 5축을 **하나도 안 바꿔도** 감도는 이미 다른 것이다.
- 굳이 5축 안에서 답하면 **최소 고정 0개.** 단 각 축의 변주는 "정체 성분만 바꾸고 감도 성분은 대역 안에 머문다"는 제약을 받는다 — 컬러를 바꿔도 채도 대역·콘트라스트 커브는 유지, 포즈를 바꿔도 카메라 거리·샷 사이즈 대역은 유지.

**해소해야 할 `excludes`:** 스타일 라벨은 조명·그레이딩·카메라 기본값을 함께 물고 오므로, "스타일 축을 바꾼다"와 "감도 앵커를 고정한다"가 같은 축에 권한자를 둘 만든다. 우선순위 1줄을 프롬프트에 그대로 넣어 해소한다:

> 스타일 라벨은 의상·실루엣·소품·태도 어휘만 공급한다. 조명·그레이딩·렌즈 성격·프레임 정책은 레퍼런스 앵커가 권한자다.

### 5.4 변주 강도 다이얼 — 3단

| 단 | 고정 | 푸는 것 | 결과 | 허용 레퍼런스 출처 |
|---|---|---|---|---|
| **D1 인접 변주** | S1~S4 전부 + 인물 정체성 + 세트 + 의상 아이템 + 팔레트 HEX | I1 포즈, I2 시선, I3 손, I7 크롭(샷 사이즈 1단계) | 같은 촬영의 다른 프레임 | **자기 자산 전용.** 외부 레퍼런스 금지 |
| **D2 자매 컷** | S1~S4 + 편집 태도 | 사용자 5축 전부 + 세트·소품, 팔레트는 관계만 유지 | 같은 감도의 **다른 작품** | 익명 무드보드, 장르 일반, 다수 작품 평균 |
| **D3 재해석** | 감도 축 2~3개만 | 나머지 전부(카메라 거리 정책·프레임 정책·장르·매체 포함) | 감도가 "영향"으로만 남음 | 식별 가능한 사진가 작품, 특정 브랜드 캠페인 |

- **D1은 복제 위험이 가장 높은 단이다.** 외부 레퍼런스에 D1을 걸면 사실상 그 작품의 아웃테이크를 만드는 것이다.
- **D2가 외부 레퍼런스의 기본값이다.** 판정 문장: 나란히 놓았을 때 "같은 사람이 찍었네"는 나오되 "이거 그 컷이네"는 안 나온다.
- 실존 인물이 등장하는 작품 → **D3 + I9 얼굴 축 전면 재작성.**
- 레퍼런스 3장 이상을 섞어 감도를 평균내면 **한 단계 완화된다** — 여러 작품에 공통으로 남는 것이 정의상 감도이고, 한 작품의 정체는 평균에서 소거되기 때문이다.

### 5.5 하드라인 — 기존 철칙 위에 얹는 증축분

[compiler.md](compiler.md) 철칙 8(실재 상표·실존 인물 참조 금지, 가상 브랜드/페르소나만) 위에 이 레인이 더하는 것이다.

**L1. 얼굴 기하 — 레퍼런스는 I9의 권한자가 아니다.** 레퍼런스에 실존 인물이 있으면 얼굴을 EVIDENCE에서 제외한다. 정체성 축 권한자는 가상 페르소나이고, `~를 닮은` 같은 닮은꼴 유도 어휘는 쓰지 않는다.

**L2. 읽히는 상표·로고·워터마크.** 레퍼런스에 있어도 산출물로 옮기지 않는다. 부정문이 아니라 긍정형 마감으로 처리한다 — `clean, brand-free, copy-free finish`, 간판은 추상적 빛 형태로.

**L3. 트레이드 드레스 — 한 레퍼런스에서 팔레트 축과 레이아웃 축을 동시에 가져오지 않는다. 하나만.** 개별 요소는 아이디어이지만 시그니처 조합(고유 컬러 락 + 고유 그리드 + 고유 타이포 처리 + 로고 위치)은 출처 표시로 기능할 수 있다.

**L4. 조합이 선을 만든다.** 개별 축은 자유롭게 가져올 수 있지만 정체 축이 여러 개 동시에 일치하면 그 조합 자체가 표현이 된다. 운영 임계는 §5.6 게이트의 **정체 축 동시 일치 ≤ 2**다.

**L5. 고유명 금지 — 실무적으로 가장 중요한 항목.** 프롬프트에 사진가 이름·브랜드명·캠페인명·특정 작품 제목을 쓰지 않는다(`in the style of [사진가]`, `[브랜드] campaign look` 포함). 관측 결과로 환원한다. 이유 셋: ① 이름은 화면 결과를 아무것도 정하지 못하는 저신호 어휘다(철칙 3) ② **이름 하나가 정체 축 여러 개를 한꺼번에 끌어와 다이얼을 무력화한다** — D2로 설정해도 결과는 D1이 된다 ③ 산출물이 아니라 프롬프트 자체가 의도의 증거로 남는다.

**L6. 인식 가능한 실재 장소·행사·매장.** 지역명·매장명 대신 건물 재료·거리 폭·간판 처리·시간대 빛·팔레트로 환원한다. 랜드마크가 그 컷의 I5면 대체한다. 처리 정본은 [editorial/tier2-safety.md](editorial/tier2-safety.md).

**L7. 원본을 산출물에 동봉하지 않고, 나란히 배치해 홍보하지 않는다.** `feeds` 인터페이스로 넘어가는 것은 관측 사실이고 원본 파일 경로·파일명은 넘어가지 않는다. side-by-side 비교는 분쟁에서 원고가 쓰는 표준 증거 형식이다.

### 5.6 사후 게이트 — 제3자가 결과물만 보고 판정

프롬프트도 의도도 없이, 두 이미지를 나란히 놓고 센다.

**정체 축 8 — 각각 일치/불일치 이진 판정. 애매하면 일치로 센다(보수적).**

| | 축 | |
|---|---|---|
| I1 | 포즈 관절 조합 | ☐ |
| I2 | 시선 방향 + 표정 조합 | ☐ |
| I3 | 손 위치·제스처·소지물 | ☐ |
| I4 | 소품 종류와 프레임 내 배치 | ☐ |
| I5 | 배경 공간 정체 | ☐ |
| I6 | 의상 아이템 정체 | ☐ |
| I7 | 크롭·피사체 프레임 내 좌표 | ☐ |
| I8 | 렌더 텍스트 문자열·타이포 배치 | ☐ |

**감도 축 4 — 각각 유지/이탈.** S1 조명 방향·질·비율 · S2 렌즈 성격·거리·심도 · S3 그레이딩 · S4 프레임 정책.

| 판정 | 조건 | 조치 |
|---|---|---|
| **PASS (자매 컷)** | I 일치 ≤ 2 **그리고** S 유지 ≥ 3 | 통과 |
| **감도 소실** | S 유지 ≤ 2 | 복제는 아니나 목적 미달 — 재작업 |
| **복제 위험** | I 일치 ≥ 3 | S가 몇이든 재작업 |
| **레드라인** | 아래 중 하나라도 — 개수 무관 | 폐기 |

레드라인 4종: ① I5가 실재 랜드마크·실재 매장과 일치 ② 화면에 읽히는 실재 로고·워터마크 ③ 실존 인물 얼굴 유사 ④ 캠페인 시그니처 조합(팔레트 + 레이아웃 + 타이포 처리가 동시에 특정 캠페인과 일치).

단계별 목표값: **D1** I 5~6 / S 4 · **D2** I ≤ 2 / S 3~4 · **D3** I 0 / S 2.

### 5.7 언어 테스트 — 계량 게이트의 보조

제3자에게 두 문항을 묻는다.

- **Q1** "같은 사람이 만든 것 같은가?" → **예**여야 한다(감도 유지).
- **Q2** "원본을 본 사람이 '이거 그 컷이다'라고 지목하는가?" → **아니오**여야 한다(복제 회피).

| Q1 / Q2 | 판정 |
|---|---|
| 예 / 아니오 | 통과 |
| 예 / 예 | 복제 |
| 아니오 / 아니오 | 실패한 변주(무관한 이미지) |
| 아니오 / 예 | 모순 — 재판정 |

**엄격판 Q2′:** 원본을 본 적 없는 제3자에게 산출물만 보여주고 "참고했을 원본을 검색해서 찾아올 수 있는가"를 묻는다. 찾아오면 넘은 것이다.

## 6. 컴파일 타깃 — 관찰 하나가 셋을 먹인다

관찰 레코드는 하나이고, `feeds` 엣지도 하나이고, 컴파일만 타깃 수만큼 생긴다([prompt-graph.md](../prompt-graph.md) §5). 타깃마다 레코드를 다시 만들지 않는다.

**타깃 기본값:** 이 레인에서 타깃이 지정되지 않으면 **gpt-image-2**로 컴파일한다. 목적축이 명시되면 [model-routing.md](model-routing.md)가 이긴다. **Higgsfield는 모델 id로 지정되어 들어온다**(`soul_2`·`nano_banana_pro`·`seedream_v5_pro` 등) — 그 지정 자체가 라우팅 결정이므로 다시 라우팅하지 않는다.

### 6.1 타깃 대조표

| 축 | **gpt-image-2** (기본) | **Higgsfield 모델 id** | **Midjourney** |
|---|---|---|---|
| 표면 | S1 스키마 / S2 `gpt_image_2` / S3 | S2 | S3 |
| 길이 | 32,000자 상한 — 실질 제약이 없어 **신호 밀도가 지배** | 계약 상한 없음 → **신호 밀도** | **단어 수** 40 권장 · 60 경고 · 80 초과 금지 |
| 네거티브 | 별도 필드 없음 → **긍정형 재서술** | 이미지·영상 생성 모델에 `negative_prompt` 없음 → **긍정형 재서술** | `--no` **인라인**, 단일 명사 쉼표 나열 |
| 비율 | S1은 스키마 enum, S2는 모델의 `aspect_ratios` | 모델의 `aspect_ratios`. **빈 배열이면 넘기지 않는다** | `--ar 정수:정수` (소수점 불가) |
| 레퍼런스 전달 | `image` 롤(S2) / S1은 계약 필드 | `medias.roles` — 확신 없으면 `image` | 슬롯이 갈린다(§6.4) |
| 조용한 실패 | — | `aspect_ratio` 근사 치환·파라미터 기본값 주입 | 모더레이션이 `--no` 값을 단어 단위로 읽음 |

길이 상한은 **세 층에서 오고 가장 좁은 것이 이긴다:** 전달 채널(텔레그램·hermes 2000자 / Claude Code·앱 실질 무제한) · 타깃 엔진(위 표) · 기계 계약(`prompt-bundle/v1` = 2000, 스키마 강제). **전역 2000자 하드라인은 없다.** 붙여넣기 UI 표면의 2000자는 S3에만 적용되는 계약이다([surfaces.md](surfaces.md) §3).

### 6.2 산출 형식 — 언제나 한 블록

복사 한 번으로 끝나야 한다. **네거티브도 그 블록 안에 있다.** 다중 컷일 때만 컷당 1블록으로 나눈다(리프 하나 = 블록 하나).

네거티브는 엔진의 형태에 따라 세 갈래다.

| 엔진의 네거티브 형태 | 산출 |
|---|---|
| 인라인 문법(Midjourney `--no`) | 블록 안에 그대로 |
| 별도 API 필드만 존재 | 같은 블록 안 **라벨 꼬리**로 — 두 번 복사시키지 않는다 |
| 없음(Higgsfield 이미지·영상 전 모델, gpt-image-2) | **긍정형 재서술** |

`Negative:` 라벨 섹션을 별도 블록으로 떼는 것은 전 티어 금지다([compiler.md](compiler.md) §2 철칙 2).

### 6.3 gpt-image-2

- 상한이 크므로 길이가 아니라 **신호 밀도**가 게이트다. 관찰 레코드에서 `observed`/`inferred`만 실리므로 분량은 저절로 통제된다. 상한을 채우려고 장식을 늘리지 않는다.
- S1 기계 핸드오프로 나가면 `ar`·`size`·`quality`는 **스키마 enum이 전부**이고 `python3 contracts/validate.py`를 실제로 통과해야 한다.
- S2(Higgsfield의 `gpt_image_2`)로 나가면 비율·해상도가 그 모델의 런타임 정의를 따른다. 파라미터로 되는 축은 산문에 중복하지 않는다.

### 6.4 Higgsfield 모델 id

- **모델의 `parameters` / `aspect_ratios` / `medias.roles`가 계약이다.** 런타임 확인이 문서보다 우선한다.
- **롤 강제는 비대칭이다** — `image_references`를 선언한 모델에 `image`를 주면 서버가 자동 교정하고, `image`를 선언한 모델에 `image_references`를 주면 하드 에러다. 확신 없으면 `image`를 쓴다. (2026-07-25 런타임 프로브)
- **`aspect_ratio`는 에러가 아니라 조용히 근사 치환된다.** 파라미터 기본값도 마찬가지로 주입된다. 판형이 계약인 작업에서는 `get_cost:true` 프리플라이트로 `adjustments`를 읽는다.
- **롤이 전달하는 것을 프롬프트에서 다시 묘사하지 않는다.** 레퍼런스가 이미 넘긴 축을 산문으로 반복하면 권한자가 둘이 된다.
- 정체성 고정 경로 셋(학습형 `soul_id` / 단일 `image` 롤 / 프롬프트 내 Element 플레이스홀더)은 **모델 집합이 서로 배타적이다.** 재컴파일 대상이 어느 경로로 만들어졌는지가 이식 가능 범위를 정한다.
- 모델 후보·로스터 스냅샷·신선도 규칙은 [model-routing.md](model-routing.md) 소관이다.

### 6.5 Midjourney

- **길이는 문자 수가 아니라 단어 수로 잰다.** 40 권장 / 60 경고 / 80 초과 금지. 관찰 레코드를 그대로 옮기면 반드시 넘으므로, 감도 축 4개를 각각 한 구로 압축하고 정체 축은 다이얼이 허용한 것만 남긴다.
- **`--no`는 단일 명사만 쉼표로 나열한다.** 모더레이션이 `--no`의 모든 단어를 독립적으로 읽으므로 다어절 구는 오탐을 만든다(`--no modern clothing` → "no clothing"). (공식 문서 확인)
- **레퍼런스 슬롯이 무엇을 가져오는지가 다르다.** Image Prompt = 내용·구도·색 / `--sref` = 감도(색·매체·질감·조명) / 무드보드는 **`--p`로 참조한다**(`--sref`가 아니다). 복제 회피의 1차 레버는 **구도·의상을 Image Prompt에서 `--sref`로 옮기는 것**이다.
- **subject leakage는 Midjourney가 인정한 공식 용어다.** V7 sref 재작성으로 완화됐으나 제거되지 않았다. 누출이 남으면 ① 텍스트에서 피사체·의상을 과잉 특정해 레퍼런스 의상이 들어올 자리를 없앤다 ② 레퍼런스를 클로즈업에서 환경 샷으로 교체한다(포트레이트가 더 많이 샌다).
- **무드보드에서는 `--sw`가 조용히 무시된다**(비호환). 강도 다이얼은 `--stylize`다. (공식 문서 확인)
- 텍스트 프롬프트는 "레퍼런스를 어떻게 고쳐라"가 아니라 **원하는 장면을 독립적으로 완결되게** 쓴다. 이것이 §5.3의 "정체 성분만 바꾼다"를 Midjourney 표면에서 실행하는 방법이다.
- 문법·파라미터의 정본은 Midjourney 문법 레퍼런스 소관이다. 이 파일은 from-image 레인이 쓰는 레버만 다룬다.

### 6.6 `feeds` 인터페이스 — 관찰 → 컴파일

| 항목 | 내용 |
|---|---|
| 넘어가는 것 | 축별 값 + 상태 + confidence. `intent`/`signature` 태그(§4.1) |
| 넘어가지 않는 것 | 원본 파일 경로·파일명, `ref-NN` 라벨, 원시 비전 payload, `artifact` 태그가 붙은 값, `unreadable`/`not_visible`/`not_applicable` 값 |
| 받았는지 판정하는 법 | 산출 블록의 각 문장이 레코드의 **어느 축에서 왔는지 역추적 가능**하다. 역추적되지 않는 문장은 관찰이 아니라 창작이다 |

## 7. 출력 게이트 체크리스트

### 7.1 관찰 단계

- [ ] 첨부 전부에 대해 레코드가 존재한다. 원시 payload가 문자열 처리기에 들어가지 않았다
- [ ] 모든 값에 상태 5종 중 하나 + confidence + 픽셀 근거가 붙어 있다
- [ ] `unreadable`/`not_visible`을 `observed`로 승격한 값이 0개다
- [ ] 감도 축이 형용사가 아니라 결과 어휘 + 수치로 적혀 있다
- [ ] 의상이 하드락이면 의상 분석 계약으로 위임했고, 스키마를 여기서 다시 쓰지 않았다

### 7.2 생성물 재컴파일일 때만

- [ ] 모든 값에 `intent`/`signature`/`artifact` 태그가 붙어 있다
- [ ] `artifact` 값이 프롬프트에 0개다. 부정문으로 배제하지도 않았다
- [ ] `signature`가 같은 모델 재실행이면 빠졌고, 다른 모델 이식이면 결과 어휘로 재서술됐다
- [ ] I8 렌더 텍스트를 관찰에서 그대로 전사하지 않았다. 카피는 사용자 확인을 거쳤다
- [ ] 원래 프롬프트 존재 여부를 물었다. 있으면 그것이 권한자다

### 7.3 복제 회피 (프롬프트 측 사전 게이트)

- [ ] 레퍼런스 EVIDENCE가 **어느 축의 권한자인지 명시**되어 있다(고아 EVIDENCE 아님)
- [ ] 외부 레퍼런스가 권한을 갖는 축이 S1~S4뿐이다. 정체 축에 권한이 있으면 그건 D1이며 자기 자산에만 허용된다
- [ ] 사진가·브랜드·캠페인·작품 고유명이 프롬프트에 0개다
- [ ] I9 얼굴 기하의 권한자가 가상 페르소나이고 레퍼런스가 아니다. 닮은꼴 유도 어휘 0개
- [ ] 한 레퍼런스에서 팔레트 축과 레이아웃 축을 동시에 가져오지 않았다
- [ ] HEX를 통짜 복사하지 않고 관계만 가져와 재생성했다(D1 제외). 조명 수치는 그대로여도 된다
- [ ] 스타일 라벨 ↔ 감도 앵커의 권한 충돌이 우선순위 1줄로 해소되어 있다
- [ ] 원본 파일 경로·파일명·`ref-NN` 라벨이 산출 블록에 0개다

### 7.4 산출 형식

- [ ] 블록이 하나다. 네거티브가 그 안에 있다. 다중 컷이면 컷당 1블록이다
- [ ] 네거티브 형태가 엔진의 3분기 중 맞는 갈래다
- [ ] 길이가 **세 층 중 가장 좁은 상한** 안이다. Midjourney면 단어 수로 쟀다
- [ ] 파라미터로 표현되는 축이 산문에 중복되지 않았다
- [ ] Higgsfield면 롤·비율이 그 모델의 런타임 정의 안이고, 판형이 계약이면 `adjustments`를 확인했다
- [ ] 산출의 각 문장이 레코드의 어느 축에서 왔는지 역추적된다

### 7.5 생성 후

- [ ] §5.6 스코어카드로 판정했고 PASS 조건(I ≤ 2 그리고 S ≥ 3)을 만족한다. 레드라인 0건
- [ ] §5.7 Q1/Q2가 예/아니오다
