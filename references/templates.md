# 모드별 골격 + 채워진 예시 + 심화 규칙

골격은 프롬프트를 채우기 위한 의무 양식이 아니다. 결과를 바꾸는 슬롯만 남기고, 내용 없는 슬롯은 출력하지 않는다.

## 범용 조립 규칙

| 슬롯 | 포함 조건 | 생략 조건 |
|---|---|---|
| 역할 | 전문성·권한이 결과를 바꿀 때 | 일반 실행자로 충분할 때 |
| 목표·결과 | 항상: 끝난 상태를 판정해야 할 때 | 없음 |
| 입력·컨텍스트 | 답을 바꾸는 사실·경로·원문이 있을 때 | 장식 설명뿐일 때 |
| 완료 기준 | 제3자 판정·품질 증거가 필요할 때 | 단순 즉시 응답에 이미 명확할 때 |
| 제약·게이트 | 실제로 열려 있고 환경이 막지 않으며 발생 가능한 위험일 때 | 세 조건 중 하나라도 아닐 때 |
| 절차·도구 | 순서·도구 계약이 실패를 막을 때 | 방법을 과도하게 고정할 때 |
| 출력 형식 | 소비자가 형식·스키마를 요구할 때 | 평문이 더 명확할 때 |
| 검증·정지 | 실행·조사·반복 작업의 완료 또는 blocked 판정이 필요할 때 | 한 번의 저위험 응답일 때 |

**내용 없는 슬롯은 출력하지 않는다.** 이는 게이트 필요성 테스트의 슬롯판이다. 단순 저위험 과제는 헤딩 없는 1~3문장 명령형이 정답일 수 있다. 가독성이 인위적 압축보다 우선한다. 결론과 핵심 제약을 앞에 두고, 구조는 정보가 생길 때만 추가한다.

채워진 예시 (256자 실측):

```text
매주 금요일 18:00, ~/reports/weekly/의 신규 CSV(헤더 date,team,metric,value)를 팀별 요약표로 변환해 ~/reports/summary.md에 추가하라. 이미 처리한 파일은 건너뛴다(재실행 시 중복 행 0). 파싱 오류 파일은 삭제하지 말고 건너뛴 뒤 ~/logs/weekly.log에 사유를 남긴다. 완료 기준: 처리 파일 수·추가 행 수가 로그에 1줄 기록되고 summary.md 행 수가 로그 누적 수와 일치한다.
```

## 슬롯 자동 채움 — 빈칸은 묻기 전에 채운다

추론 순서는 대화 컨텍스트 → 언급된 저장소·파일 증거 → 아래의 저위험 기본값 → 결과나 하드라인을 바꾸는 빈칸만 질문 ≤3이다. 채운 값은 `가정:` 노트에 남긴다. 결과 자체가 없으면 슬롯 채움이 아니라 요구 구체화 경로다.

| 모드 | 비었을 때 채움 규칙 |
|---|---|
| GOAL | 마지막 언급 프로젝트를 대상에 쓰고, 없으면 행동 전 대상을 식별·보고하게 한다. 완료 증거는 관련 테스트·명령 출력으로 둔다. |
| TEAM | 목표·범위는 대화에서, 레인은 파일·검증 표면 경계로 2~3개, 검증은 기존 테스트나 측정기로 채운다. |
| CONTRACT | 결과·DoD는 산출물 관례와 대화에서, 검증은 실제로 열거나 실행하는 최소 절차로 둔다. |
| BUSINESS | 직무·현실 제약·형식은 국내 일반 실무 관례에서 채우되 결과를 가르는 법령·기준 버전은 가정 또는 질문으로 처리한다. |
| MODEL | 대상이 미상이면 이식 가능한 역할·목표·제약·컨텍스트·형식·검증 구조를 쓴다. |
| 리서치 | 출처 유형·확인 날짜·반대 증거는 시점과 결론을 바꿀 때 추가한다. |
| 추출 | 스키마 밖 정보는 추가하지 않고 원문에 없는 필드는 null로 둔다. |

**추론 불가 슬롯 — 질문이 정당한 유일 목록 (정본)**: 퇴고·델타의 기준 원문, 합성할 원본 이미지, 정확한 렌더 카피·브랜드명, 실계정·실서비스 식별자, 결과를 가르는 비저위험 양자택일 방향. 이 밖의 빈칸으로 먼저 묻는 것은 질문 반사다.

## 수신자·채널 적응

기본은 도구를 쓸 수 있는 자율 실행자에게 메신저 한 메시지로 전달하는 경우다. 파일·도구 없는 대화 UI에는 필요한 자료를 인라인하고, API 시스템 프롬프트에는 입력 위치·불확실 시 행동을 명시한다. 사람 팀원에게는 사람용 제목·불릿으로 바꾸되 결과·경계·검증은 유지한다. 채널이 달라도 블록당 2000자 계약은 유지하며, 다른 파일시스템에는 경로 참조 대신 자기완결로 쓴다.

## GOAL — 자율 에이전트 goal 지시문

**필수 코어**: 결과, 완료 기준, 최종 보고, 막혔을 때. **조건부 확장**: 게이트·컨텍스트·우선순위는 실제 정보가 있을 때만 추가한다. 판정자가 앞부분과 최종 보고를 본다고 가정해 핵심을 앞배치한다.

```text
<결과 — 끝난 상태>

완료 기준:
- <경로·명령 출력·개수 등 판정 증거>

최종 보고: 변경한 것 / 검증한 것(명령+결과) / 안 한 것과 이유.
막히면: <기준> → 시도한 것/막힌 지점/필요한 것 형식으로 종료.
```

리서치+구현, 광역 리팩터링, 승인 대기, 장기 관찰, 독립 결과 둘 이상은 각각 별도 goal로 나눈다. 승인 행동은 증거와 승인 요청문 준비까지를 현재 goal의 종료로 둔다.

채워진 예시 (215자 실측):

```text
~/src/acme-api 의 플래키 결제 테스트 2건이 원인 수정으로 연속 20회 통과하는 상태.

완료 기준:
- pytest tests/payment --count 20 연속 통과 로그
- 원인과 수정 diff 요약 1문단

최종 보고: 변경한 것 / 검증한 것(명령+결과) / 안 한 것과 이유.
막히면: 2시간 내 재현 불가 → 시도한 것/막힌 지점/필요한 것 형식으로 종료.
```

## TEAM — 멀티에이전트 킥오프

**필수 코어**: 목표, 범위·비목표, 레인, 검증, 산출물. **조건부 확장**: 컨텍스트·경계·상세자료는 워커 판정을 바꿀 때만 넣는다. 같은 acceptance를 공유하는 구현과 검증은 같은 레인에 둔다.

```text
목표: <구체적 결과 하나>
범위: <대상> / 비목표: <명시적 제외>
작업 방식: <독립 검증 표면별 레인과 조율 규칙>
검증: <명령·시나리오·증거>
산출물: <파일·패치·보고서>
```

채워진 예시 (255자 실측):

```text
목표: acme-web 초기 로드 p95 를 3.2s에서 1.5s 이하로 내리는 패치 완성.
범위: ~/src/acme-web 로컬 브랜치 / 비목표: 디자인 변경, 백엔드 수정.
작업 방식: prime이 병목 인벤토리 확정 후 번들·폰트 레인을 분배하고, 같은 acceptance의 구현+검증은 한 레인에 둔다.
검증: scripts/perf.sh 3회 중앙값 p95 ≤ 1.5s, npm test 전체 통과.
산출물: 로컬 브랜치 + 수치 비교 보고서.
```

### TEAM 역할·분해

역할·capability·분해·join·blocker·증거 규칙의 정본은 [model-playbooks.md](model-playbooks.md)다. 아래는 TEAM 골격의 `작업 방식` 필드에 넣는 압축 블록이며, 호출 접두어와 실제 모델 선택은 [adapters.md](adapters.md)에서 채운다.

```text
작업 방식: prime이 topology-first로 최상위 결과 목록과 acceptance를 확정한 뒤, 검증 표면이 독립적인 slice만 worker에 배정한다. 같은 acceptance를 공유하는 구현+테스트는 한 레인에 둔다. worker는 target/scope/non-goals/acceptance 안에서만 실행하고, prime은 모든 worker 산출물과 critic 리뷰가 같은 frozen artifact 기준으로 도착한 뒤 최종 claim을 낸다. human-only blocker만 질문한다.
```

## CONTRACT — 작업지시·시스템 프롬프트·자동화 잡

**필수 코어**: 결과, 완료 기준, 검증. **조건부 확장**: 자료·자율성·게이트·막힘 규칙·컨텍스트는 필요할 때만 붙인다. 자동화 잡은 멱등성과 오류 기록을 추가하고, 시스템 프롬프트는 DoD를 응답 품질 기준으로 바꾼다.

```text
## 결과
<끝난 상태>

## 완료 기준
- <판정 가능한 증거>

## 검증
- <실행 또는 대조 절차>
```

## BUSINESS — 직무 업무 프롬프트

**필수 코어**: 결과, 형식, 검증. **조건부 확장**: 역할·전문성, 현실 제약, 목표 수량, 작업 단계, 근거 요구는 난이도와 이해관계가 높을 때만 넣는다. 형식의 채워진 예시는 비자명하거나 비표준일 때만 추가한다. 난이도 1~2는 자연어 1~3문장, 3~5는 필요한 슬롯만, 6~7은 예외·범위 잠금·근거를 늘린다.

채워진 예시 (137자 실측):

```text
개인 카페의 메뉴별 원가율을 계산하라. 표는 메뉴 | 판매가 | 재료비 | 원가율 | 판정 | 대안으로 출력하고, 제공되지 않은 값은 추정으로 표시한다. 원가율 40% 초과 메뉴마다 가격 또는 레시피 대안 하나를 제시한 뒤 계산 근거를 대조하라.
```

## MODEL — 퇴고·델타·리서치·추출·체인

**필수 코어**: 대상 또는 원문, 결과 형식, 검증. **조건부 확장**: 역할·도구·증거 규칙은 대상 작업이 요구할 때만 쓴다. 퇴고는 수정본과 실제 변경 노트만 낸다. 델타는 지정 축만 바꾸고 기준 원문이 없으면 요청한다.

### 리서치

**필수 코어**: 조사 질문, 출처·날짜가 있는 결론 형식, 검증. **조건부 확장**: 반대 증거·불확실성은 결론을 바꿀 수 있을 때 추가한다. 공식·1차 자료를 최신 블로그나 기사보다 우선하며 상충은 숨기지 않는다.

채워진 예시 (221자 실측):

```text
## 결과
국내 클라우드 백업 3사의 요금제 비교 보고서를 작성한다.

## 완료 기준
- 각 사의 요금제명·월 요금·용량·초과 과금을 표 하나에 정리
- 모든 값에 출처 URL과 확인 날짜를 붙이고 확인 못 한 값은 "미확인"으로 표시
- 상충 수치는 병기하고 공식 가격 페이지를 우선

## 막혔을 때
가격 페이지 접근이 두 번 실패하면 해당 사를 "확인 불가"로 표기하고 나머지를 진행한다.
```

### 추출

**필수 코어**: 입력, 스키마, 원문 밖 추측 금지. **조건부 확장**: 중복 규칙·정규화는 소비자가 요구할 때만 넣는다.

채워진 예시 (130자 실측):

```text
아래 텍스트에서 다음 스키마로만 추출하라.
스키마: {"업체명": str, "연락처": str|null, "주소": str|null}
원문에 없는 필드는 null로 두고, 스키마 밖 필드·설명·주석은 출력하지 마라.
텍스트: <원문>
```

### 체인

단계별 블록을 분리하고, 앞 단계의 산출 형식과 다음 단계의 입력이 같은 스키마임을 예시로 대조한다. 각 단계에 다음 단계 전달 전 검증 한 줄을 둔다.

## DESIGN 오버레이

실행 모드의 코어 위에만 겹친다. **필수 코어**: 방향 잠금, 시각 테제, 뷰포트 검증. **조건부 확장**: 시그니처·마이크로 인터랙션·기술 제약은 구현 결과를 바꿀 때만 둔다. 방향은 제품 맥락에서 하나를 가정으로 선언하고, 막연한 미사어 대신 표면·서체·간격·상호작용처럼 관찰 가능한 속성으로 쓴다. 기존 UI 수리는 결함을 한 문장으로 지목하고 최소 수정부터 수행한다.

채워진 예시 (198자 실측):

```text
## 디자인 방향
- 방향: raw terminal — 종이 같은 오프화이트와 잉크 블랙, 정보 밀도는 높고 계층은 분명하게.
- 시그니처: 가로 시간선이 섹션 구분을 맡는다.
- 하드 제약: 기존 프레임워크와 접근성 대비를 지킨다.

## 완료 기준
- 구현 전 시각 테제 3줄을 정하고 결과와 대조
- 데스크톱과 375px에서 긴 문자열의 넘침을 확인
```

## 모호어 치환

| 모호어 | 치환 방향 |
|---|---|
| 적절히 처리해 | 실패 기준과 재시도·기록 행동을 쓴다 |
| 잘 정리해 | 표·요약의 개수와 길이를 쓴다 |
| 최대한 자세히/빨리 | 항목당 근거와 시간 제한을 쓴다 |
| 필요하면 | 조건 C가 참이면 X로 바꾼다 |
| 등, 기타, 알아서 | 목록을 확정하거나 전부임을 선언한다 |
| 중요한 것 위주로 | 판단 기준을 쓴다 |
| 완벽하게/깔끔하게/고급스럽게 | 판정 가능한 기준과 증거로 바꾼다 |
| nicely / properly / in detail | 영어 동족어도 같은 방식으로 전부 치환한다 |

## 워크플로우

1. **분류** — 모드 선택. 빈 슬롯은 §슬롯 자동 채움으로 채우고 `가정:`에 기록한다.
2. **결과 명세** — 끝난 상태 1~2문장과 자료 인벤토리. 언급된 경로는 실제로 읽고 사실을 지어내지 않는다.
3. **계약층** — 완료 기준을 게으른 보고가 통과하지 못하게 조인 뒤, 자율성·게이트·검증·에스컬레이션 중 필요한 것만 붙인다.
4. **구조층** — §범용 조립 규칙으로 슬롯을 고르고, 비자명·비표준 형식에만 채워진 예시 1개를 둔다.
5. **표면층** — 대상 모델·수신자에 맞춘다([model-playbooks.md](model-playbooks.md)). 역할 없는 단어는 지운다.
6. **감량** — 모호어 치환, 제약 모순 해소, 방법 문장 다이어트("지우면 실행자가 틀리는가?").
7. **길이 게이트** — 블록당 문자 수 실측. 초과는 SKILL.md §길이 계약대로 처리한다.
8. **드라이런** — 현실 입력 2~3개(엣지 1개 포함)로 머릿속 실행해 구멍을 수정한 뒤 출력한다.

## 안티패턴

단계 마이크로매니징(→ 결과·검증만 계약) / 장문 인라인 덤프(→ 경로 참조) / 전면 read-only 과잉 차단(→ 좁은 게이트+대체 행동) / 환경 훅이 이미 막는 행동의 이중 게이트(→ 필요성 테스트로 제거) / 질문 반사(→ 추론·기본값 먼저, 질문은 추론 불가 슬롯만) / "알아서" 위임(→ 자율성 3단 경계) / 하나의 비대 goal(→ 분할) / 예시-규칙 충돌 방치(→ 대조 감사) / 무대 지정 어휘("최고 수준으로" → 수치·구체 기준) / 패딩용 문서.

## 보강 루프

결과·모드·레인이 유지되고 한 축만 바뀌면 패치한다. 하나라도 바뀌거나 계약 요소 여러 개가 흔들리면 재작성한다. 압축은 장식 → 중복 → 방법 설명 순으로 줄이며 결과·완료 기준·게이트·검증은 제거하지 않는다. 실행 실패는 결과, DoD, 자율성, 게이트, 검증, 에스컬레이션 중 결함 요소 하나를 찾아 그 요소만 고친다. 보강 뒤에도 길이·모호어·형식·검증 계약을 다시 확인한다.
